## 📋 流程概览

```
┌─────────────────────────────────────────────────────────────────┐
│                    Issue 完整生命周期                            │
└─────────────────────────────────────────────────────────────────┘

  发现/提出          审查确认          开发修复           QA 验证
     ↓                 ↓                 ↓                  ↓
  ┌─────┐          ┌─────┐          ┌─────┐           ┌─────┐
  │ 新建 │ ───────→ │ 审查 │ ───────→ │ 开发 │ ────────→ │ 验证 │
  │Issue│          │分类 │          │修复 │           │关闭 │
  └─────┘          └─────┘          └─────┘           └─────┘
     │                │                 │                  │
     │                │                 │                  │
  任何人          Team Lead        开发者            qa-tester
  团队成员        + qa-tester                        + 报告人
```

---

## 🎭 角色与职责

### 报告人 (Reporter)
- 发现 Bug 或提出功能需求
- 创建 Issue 并填写完整信息
- 配合开发者澄清问题
- 参与最终验证

### Team Lead
- 审查新 Issue 的有效性和优先级
- 分配 Issue 给合适的开发者
- 跟踪进度，解决阻塞

### 开发者 (Developer)
- 分析 Issue 并制定解决方案
- 实现修复或功能
- 更新 Issue 状态
- 请求 QA 验证

### qa-tester
- 参与 Issue 审查（优先级评估）
- 执行验证测试
- 更新验证状态
- 关闭已验证的 Issue

## 📝 流程详解

### 阶段 1: Issue 创建

**触发条件**: 任何人发现 Bug 或有功能需求

**操作步骤**:
1. 搜索现有 Issue，确认没有重复
2. 点击 "New Issue"
3. 选择合适的模板：
   - 🐛 Bug 报告
   - 💡 功能请求
4. 填写完整信息：
   - 清晰的标题
   - 详细描述
   - 复现步骤（Bug）
   - 环境信息
   - 优先级自评
5. 提交 Issue

**质量检查清单**:
- [ ] 标题清晰描述问题
- [ ] 描述完整，无歧义
- [ ] Bug 包含复现步骤
- [ ] 填写了环境信息
- [ ] 自评了优先级
- [ ] 添加了相关截图/日志

---

### 阶段 2: Issue 审查 (Triage)

**负责人**: Team Lead + qa-tester  
**时限**: 新 Issue 24 小时内

**操作步骤**:
1. Team Lead 审查 Issue 内容
2. qa-tester 评估优先级
3. 确认/调整标签：
   - 优先级标签 (P0-P3)
   - 类型标签 (bug/enhancement)
   - 模块标签 (frontend/backend)
   - 状态改为 `confirmed`
4. 分配给合适的开发者
5. 移除 `triage` 标签

**决策矩阵**:

| 情况 | 行动 |
|------|------|
| 信息不完整 | 添加 `question` 标签，@报告人补充 |
| 重复 Issue | 标记 `duplicate`，链接到原 Issue，关闭 |
| 不属于本项目 | 标记 `wontfix`，说明原因，关闭 |
| 有效 Issue | 确认优先级，分配开发者，状态 `in-progress` |

---

### 阶段 3: 开发修复

**负责人**: 被分配的开发者  
**时限**: 根据优先级
- P0: 立即处理，当天完成
- P1: 24 小时内完成
- P2: 3 天内完成
- P3: 1 周内或迭代安排

**操作步骤**:
1. 开发者接受 Issue
2. 创建分支：`fix/issue-#{编号}` 或 `feat/issue-#{编号}`
3. 实现修复/功能
4. 编写/更新测试用例
5. 本地验证
6. 提交代码，创建 PR
7. PR 描述中关联 Issue：`Fixes #编号`
8. 更新 Issue 状态为 `review-needed`

**PR 检查清单**:
- [ ] 代码通过 CI/CD
- [ ] 通过代码审查
- [ ] 包含必要的测试
- [ ] PR 描述关联 Issue
- [ ] 更新相关文档

---

### 阶段 4: QA 验证

**负责人**: qa-tester  
**时限**: PR 合并后 24 小时内

**操作步骤**:
1. qa-tester 收到 `review-needed` 通知
2. 阅读 Issue 描述和 PR 变更
3. 执行验证测试：
   - 复现原问题（Bug）
   - 验证新功能（enhancement）
   - 回归测试相关功能
4. 记录验证结果

**验证通过**:
1. 更新 Issue 状态为 `verified`
2. 在 Issue 中添加验证评论：
   ```
   ✅ 已验证
   
   验证环境：[Staging/Production]
   验证时间：2026-03-21
   验证人：qa-tester
   
   测试结果：
   - [测试项 1] ✅ 通过
   - [测试项 2] ✅ 通过
   
   备注：无
   ```
3. 关闭 Issue

**验证不通过**:
1. 在 Issue 中说明失败原因
2. 状态改回 `in-progress`
3. @开发者重新修复
4. 附上测试日志/截图

## 📊 特殊情况处理

### 紧急 Bug (P0)

**流程加速**:
1. 创建 Issue 时明确标注 P0
2. Team Lead 立即审查（1 小时内）
3. 开发者立即放下手头工作处理
4. qa-tester 优先验证
5. 可先修复后补文档

**沟通方式**:
- 在团队群 @相关人员
- 电话/即时消息通知
- 每小时更新进度

### 跨模块 Issue

**处理方式**:
1. 添加多个模块标签
2. 分配主要责任开发者
3. 需要协作时 @相关开发者
4. 必要时创建子任务 Issue

### 阻塞 Issue

**处理流程**:
1. 开发者标记 `blocked`
2. 在评论中说明阻塞原因
3. @相关人员解决依赖
4. 依赖解决后移除 `blocked`，继续处理

## 📈 度量指标

### 每周报告指标

| 指标 | 计算方式 | 目标 |
|------|---------|------|
| 平均响应时间 | Issue 创建到首次评论 | < 4 小时 |
| 平均修复时间 | Issue 创建到关闭 | P0: 1 天, P1: 3 天 |
| 重新打开率 | 重新打开 Issue / 总关闭 Issue | < 10% |
| QA 验证通过率 | 一次验证通过 / 总验证数 | > 90% |

### 月度分析

- Issue 趋势图（按类型、模块）
- 优先级分布分析
- 开发者工作量统计
- 常见 Bug 类型分析

## 🎓 最佳实践

### Issue 创建
- ✅ 一个 Issue 只描述一个问题
- ✅ 标题简洁明确（< 60 字符）
- ✅ 使用模板确保信息完整
- ✅ 添加相关截图和日志

### Issue 评论
- ✅ 评论前搜索是否已有人提及
- ✅ 使用 @mention 通知相关人员
- ✅ 更新时说明最新状态
- ✅ 决策性评论要清晰明确

### 状态更新
- ✅ 状态变化时立即更新标签
- ✅ 长时间无更新时主动同步进度
- ✅ 遇到阻塞及时标记并通知
