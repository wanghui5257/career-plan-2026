## 重大挑战与解决

### 1. 服务多次崩溃（502 Bad Gateway）

**问题**：
- 服务启动后频繁崩溃
- 登录 API 返回 500 错误
- 团队等待 2.5+ 小时

**根因**：
- 旧 JAR 包（59MB）未替换
- 新 JAR 包（77MB）构建到不同目录
- 服务从旧目录启动

**解决**：
1. 验证 JAR 包大小和位置
2. 复制新 JAR 到启动目录
3. 杀死所有 Java 进程
4. 重新启动服务
5. 验证日志包含新代码

**教训**：
- ✅ 构建后验证 JAR 包大小
- ✅ 确认服务启动目录
- ✅ 部署后验证日志
- ✅ 使用 `lsof -p <PID> | grep .jar` 验证加载的 JAR

---

### 2. 密码哈希不匹配

**问题**：
- Admin 登录返回 400 错误
- 测试账号无法登录
- 密码反复被重置

**根因**：
1. 初始密码哈希不是正确 bcrypt 格式
2. 测试脚本修改密码后未恢复
3. bash  quoting 问题导致哈希截断

**解决**：
1. 使用 Python bcrypt 库生成正确哈希
2. 每个修改密码用例后立即恢复
3. 使用 `trap` 确保清理执行
4. 使用 bt 命令直接重置数据库

**教训**：
- ✅ 使用可靠工具生成 bcrypt 哈希
- ✅ 测试脚本必须包含清理机制
- ✅ 使用 `trap cleanup EXIT` 确保清理
- ✅ 测试前备份原始密码

---

### 3. P0 安全修复未部署

**问题**：
- 手动验证返回 401（已修复）
- 自动化测试返回 200（未修复）
- 团队困惑修复是否生效

**根因**：
- 测试脚本 Bug：`${7:-$TOKEN}` 空字符串回退到默认 Token
- 无授权测试实际使用了 Token
- 返回 200 是假阴性

**解决**：
1. 修复测试脚本逻辑
2. 确保无授权测试真正无 Token
3. 使用 domain 验证（不是 localhost）
4. 重新执行测试

**教训**：
- ✅ 测试脚本必须验证真正无授权
- ✅ 禁止使用回退逻辑：`${7:-$TOKEN}` → `$7`
- ✅ 手动验证 + 自动化测试双重确认
- ✅ 使用 domain 验证（与生产一致）

---

### 4. 测试脚本逻辑错误

**问题**：
- 测试通过率 61.5%（未达 90% 目标）
- 部分用例失败原因不明

**根因**：
1. 检查 HTTP 状态码（后端统一返回 200）
2. 应检查 JSON `code` 字段
3. 密码修改使用 POST 而非 PUT

**解决**：
1. 修复测试脚本检查 JSON `code` 字段
2. 修正 HTTP 方法（POST → PUT）
3. 重新执行测试

**教训**：
- ✅ 测试前确认 API 响应格式
- ✅ 使用正确的 HTTP 方法
- ✅ 检查业务状态码（JSON code），不是 HTTP 状态
- ✅ 测试脚本版本控制

---

### 5. 浏览器缓存问题

**问题**：
- 部署后功能未更新
- 测试仍显示旧 Bug

**根因**：
- 浏览器缓存旧 JS 文件
- 未清除缓存测试

**解决**：
1. Chrome: `Ctrl+Shift+Delete` → 清除缓存
2. 或：F12 → Network → "Disable cache" → `Ctrl+F5`
3. Mac: `Cmd+Shift+R`

**教训**：
- ✅ 测试前清除浏览器缓存
- ✅ 使用无痕模式测试
- ✅ 验证部署文件包含修复代码（grep）

**理由**：权限问题，需要 root 访问
**结果**：✅ 成功重置所有测试账号

**决策**：禁止 IP+端口测试，必须使用域名
**理由**：服务器端口未开放，域名代表真实用户环境
**结果**：✅ 测试环境一致，避免部署问题

## 流程改进

### 1. 部署验证流程
**改进前**：部署后直接测试
**改进后**：
```
1. 构建 → 验证产物大小
2. 部署 → 验证文件时间戳
3. 重启 → 验证服务日志
4. 测试 → 验证关键功能
5. 确认 → grep 验证修复代码
```

### 2. 测试清理机制
**改进前**：测试完成后统一恢复
**改进后**：
```
1. 测试前备份原始数据
2. 每个修改用例后立即恢复
3. 使用 trap 确保清理执行
4. 测试完成验证数据完整性
```

```
5. 审查通过后合并
```
